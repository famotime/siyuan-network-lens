# 关键配置文件说明

本文整理当前仓库中最关键的配置文件，说明它们分别负责什么、常见修改场景是什么、修改时需要注意哪些联动点。

目标不是罗列所有文件，而是帮助后续维护时快速判断:

- 这个配置该去哪里改
- 改完会影响开发、构建、发布还是插件运行时行为
- 哪些配置已经有单一来源，哪些地方需要联动检查

## 配置分层总览

当前项目的关键配置大致分成三层:

1. 工程与构建配置
2. 插件元数据与发布配置
3. 插件运行时业务配置

建议优先按下面的思路定位:

- 改命令、依赖、版本脚本: 看 `package.json`
- 改打包目标、watch 输出目录、静态资源拷贝: 看 `vite.config.ts`
- 改插件名称、版本、最低思源版本、商店展示信息: 看 `plugin.json`
- 改 TS 编译行为、路径别名: 看 `tsconfig.json`、`tsconfig.node.json`
- 改 lint 规则和代码风格: 看 `eslint.config.mjs`
- 改本地开发环境路径: 看 `.env`、`.env.example`
- 改用户在插件设置页里可配置的默认值和持久化结构: 看 `src/types/config.ts`
- 改顶部统计卡片的默认顺序、默认显隐和设置页文案: 看 `src/analytics/summary-card-config.ts`

## 一、工程与构建配置

### `package.json`

职责:

- 定义项目基础信息、依赖版本和 npm 脚本
- 作为发布版本号来源之一

当前关键脚本:

- `npm run dev`: `vite build --watch`
- `npm test`: `vitest run`
- `npm run build`: `vite build`
- `npm run release[:patch|:minor|:major|:manual]`: 执行 `release.js`

常见修改场景:

- 新增开发依赖或运行时依赖
- 调整构建、测试、发布命令
- 升级项目版本号

注意事项:

- 版本号会和 `plugin.json` 联动，发布时两边应保持一致
- `npm run build` 会更新根目录 `package.zip`

### `vite.config.ts`

职责:

- 定义 Vite 构建入口和输出方式
- 决定 watch 模式输出到思源工作区，普通构建输出到 `dist/`
- 配置静态资源拷贝和 `package.zip` 打包

当前关键行为:

- 通过 `VITE_SIYUAN_WORKSPACE_PATH` 决定 watch 模式输出目录
- 入口为 `src/index.ts`
- 产物格式为 `cjs`
- 非 watch 模式下自动生成根目录 `package.zip`
- 自动拷贝 `README*.md`、`icon.png`、`preview.png`、`plugin.json`、`src/i18n/**`
- 路径别名 `@ -> src`

常见修改场景:

- 修改打包入口、输出路径或输出格式
- 新增需要随插件一起分发的静态文件
- 调整本地 watch 开发体验

注意事项:

- 这里是“构建如何发生”的中心文件
- 如果修改了输出目录策略，需同步验证 `npm run dev` 和 `npm run build`
- 目前文件里读取了 `VITE_DEV_DIST_DIR`，但实际输出目录逻辑主要仍由 `VITE_SIYUAN_WORKSPACE_PATH` 驱动，后续如要启用自定义输出目录，应先统一这里的判定逻辑

### `tsconfig.json`

职责:

- 定义前端源码的 TypeScript 编译选项

当前关键配置:

- `baseUrl: "."`
- 路径别名:
  - `@/* -> ./src/*`
  - `@/libs/* -> ./src/libs/*`
- `allowJs: true`
- `checkJs: true`
- `strict: false`
- `noUnusedLocals: true`
- `noUnusedParameters: true`
- `noEmit: true`

常见修改场景:

- 新增路径别名
- 调整 TS 严格度
- 扩展 `include` 范围

注意事项:

- 当前不是严格模式，很多运行时约束主要依赖测试和业务逻辑控制
- 如果准备逐步收紧类型规则，建议分阶段进行，不要一次性打开全部严格选项

### `tsconfig.node.json`

职责:

- 为 Node 侧配置文件提供 TS 编译上下文
- 当前主要服务于 `vite.config.ts`

常见修改场景:

- 新增 Node 侧 TS 配置文件时，需要把它们纳入 `include`

### `eslint.config.mjs`

职责:

- 定义仓库代码规范和 lint 行为

当前关键特点:

- 基于 `@antfu/eslint-config`
- 启用了 Vue 和 TypeScript
- 忽略 `dist`、`node_modules`
- 对 Vue 单文件组件的属性换行、block 顺序等有明确约束
- 关闭了部分偏严格规则，例如:
  - `no-console`
  - `ts/explicit-function-return-type`
  - `ts/strict-boolean-expressions`

常见修改场景:

- 团队想统一风格
- 某条规则误伤过多，需要收敛
- 需要新增自定义 lint 规则

注意事项:

- 这是“代码格式和静态检查口径”的中心文件
- 如果某个风格调整影响较大，最好配套一次批量格式化和单独提交

### `.env.example` 与 `.env`

职责:

- 提供本地开发环境变量

当前关键变量:

- `VITE_SIYUAN_WORKSPACE_PATH`: 思源工作区路径，watch 构建时会输出到对应插件目录
- `VITE_DEV_DIST_DIR`: 预留的自定义输出目录变量

建议:

- `.env.example` 只保留模板，不写本机绝对路径
- `.env` 用于本地实际开发，不应作为团队共享配置说明的唯一来源

## 二、插件元数据与发布配置

### `plugin.json`

职责:

- 定义思源插件元信息
- 决定插件在思源中的名称、描述、兼容版本和展示信息

当前关键字段:

- `name`: `siyuan-network-lens`
- `version`
- `minAppVersion`: `3.5.7`
- `displayName.zh_CN`: `脉络镜`
- `description.zh_CN`
- `readme.zh_CN`: `README_zh_CN.md`
- `keywords`

常见修改场景:

- 升级插件版本
- 修改插件展示名称或描述
- 提高最低思源版本要求

注意事项:

- `version` 需要和 `package.json` 同步
- 发布前如果改了展示信息，最好也检查 `README.md` 与 `preview.png` 是否匹配当前功能

### `release.js`

职责:

- 辅助版本发布
- 自动更新 `plugin.json` 和 `package.json` 的版本号
- 自动执行 git add、commit、push、tag、push tag

当前发布模式:

- `--mode=patch`
- `--mode=minor`
- `--mode=major`
- `--mode=manual`

注意事项:

- 这是带副作用的脚本，不只是“改版本号”
- 运行后会直接尝试:
  - 提交代码
  - 推送分支
  - 创建 tag
  - 推送 tag
- 如果仓库处于脏工作区，或者当前分支/远端状态不符合预期，先不要直接运行

## 三、插件运行时业务配置

### `src/index.ts`

职责:

- 这是运行时配置的入口
- 负责加载、补全、持久化插件设置

当前关键行为:

- 启动时通过 `loadData('settings.json')` 读取用户设置
- 用 `ensureConfigDefaults` 对旧配置和缺省字段做补全
- 通过 `watch` 自动把配置保存回 `settings.json`

这意味着:

- 真正持久化到用户环境的是 `settings.json`
- `src/types/config.ts` 定义的是该持久化结构和默认值

### `src/types/config.ts`

职责:

- 定义 `PluginConfig` 类型
- 定义默认配置 `DEFAULT_CONFIG`
- 定义配置补全逻辑 `ensureConfigDefaults`

当前主要配置分组:

- 顶部统计卡片总开关: `showSummaryCards`
- 单卡片显隐:
  - `showDocuments`
  - `showRead`
  - `showReferences`
  - `showRanking`
  - `showCommunities`
  - `showTrends`
  - `showOrphans`
  - `showDormant`
  - `showBridges`
  - `showPropagation`
- 主题文档配置:
  - `themeNotebookId`
  - `themeDocumentPath`
  - `themeNamePrefix`
  - `themeNameSuffix`
- 已读规则配置:
  - `readTagNames`
  - `readTitlePrefixes`
  - `readTitleSuffixes`
  - `readPaths`
- 顶部卡片排序:
  - `summaryCardOrder`

当前兼容逻辑:

- 老字段 `showOrphanBridge` 仍被用作 `showOrphans`、`showDormant`、`showBridges` 的迁移来源

常见修改场景:

- 新增设置项
- 调整默认值
- 增加旧版本配置迁移逻辑

注意事项:

- 新增配置项时，至少要同时更新:
  - `PluginConfig`
  - `DEFAULT_CONFIG`
  - `ensureConfigDefaults`
  - 设置页 UI
  - 必要测试

### `src/analytics/summary-card-config.ts`

职责:

- 这是“顶部统计卡片配置”的单一来源
- 统一定义每张卡片的:
  - 业务 key
  - 对应配置字段
  - 默认是否显示
  - 设置页文案
  - 默认顺序
  - 旧字段迁移来源

当前默认顺序:

1. `read`
2. `orphans`
3. `ranking`
4. `documents`
5. `trends`
6. `references`
7. `communities`
8. `propagation`
9. `bridges`
10. `dormant`

当前默认显隐:

- 默认显示前三张卡片: `read`、`orphans`、`ranking`
- 其余卡片默认隐藏，用户可在设置页中手动开启

为什么这个文件重要:

- 调整默认顺序，不需要去多个文件找常量
- 调整某张卡片默认显隐，不需要改多处分支判断
- 设置面板文案和配置字段映射也来自这里

当前已接入的位置:

- `src/types/config.ts`: 生成默认显隐配置和迁移补全
- `src/analytics/summary-card-order.ts`: 生成默认顺序
- `src/App.vue`: 判断卡片是否显示
- `src/components/SettingPanel.vue`: 渲染卡片设置项

建议:

- 后续新增或删除统计卡片，优先从这里开始改
- 尽量不要再把卡片配置写回到 `App.vue` 或设置面板里做硬编码分支

### `src/components/setting-panel-data.ts`

职责:

- 负责设置页的数据准备

当前关键行为:

- 读取笔记本列表
- 从文档块中查询标签，生成已读标签选项
- 对已读相关配置做最小默认值修正

常见修改场景:

- 设置页新增依赖数据库查询的候选项
- 调整已读标签候选来源

注意事项:

- 这个文件偏“设置页数据层”，不是持久化配置定义层
- 持久化字段本身仍应回到 `src/types/config.ts`

## 四、常见修改任务应该改哪里

### 1. 想新增一个顶部统计卡片

优先修改:

- `src/analytics/summary-card-config.ts`

然后检查:

- `src/analytics/summary-details.ts`
- `src/analytics/summary-card-order.ts`
- `src/types/config.ts`
- `src/components/SettingPanel.vue`
- `src/components/SummaryCardsGrid.vue`
- 相关测试

### 2. 想只调整某张卡片默认显示或默认顺序

优先修改:

- `src/analytics/summary-card-config.ts`

### 3. 想新增一个新的设置项

优先修改:

- `src/types/config.ts`
- `src/index.ts`
- `src/components/SettingPanel.vue`

如果设置项还需要候选数据:

- `src/components/setting-panel-data.ts`

### 4. 想修改插件名称、描述、最低兼容版本

优先修改:

- `plugin.json`

如果文案也要同步:

- `README.md`

### 5. 想调整本地 watch 构建输出到哪个思源工作区

优先修改:

- `.env`

参考模板:

- `.env.example`

### 6. 想调整发布流程

优先修改:

- `release.js`
- `package.json`

## 五、维护建议

为了减少配置散落，当前建议遵守下面几条:

- “用户设置持久化结构”统一收口到 `src/types/config.ts`
- “统计卡片定义”统一收口到 `src/analytics/summary-card-config.ts`
- “插件元数据”统一收口到 `plugin.json`
- “构建行为”统一收口到 `vite.config.ts`

如果后续发现某类配置开始在多个文件里重复出现，优先考虑先抽成单一来源，再继续加功能。
