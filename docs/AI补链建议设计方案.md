# AI 补链建议设计方案

本文面向开发落地，给出“脉络镜（Network lens）”中 AI 补充链接建议能力的目标场景、总体架构、具体设计与分阶段实施方案。

文档基于当前实现现状展开，重点回答以下问题：

- AI 是否可以让补链建议更全面、更准确
- 如何在避免高延迟的前提下引入 AI
- 如何复用现有结构分析结果
- 文档摘要、关键词、向量等派生信息应如何生成与存储

## 1. 背景与现状

### 1.1 当前实现做了什么

当前“孤立文档补充链接建议”主要由以下两部分组成：

1. 建议生成
2. 建议执行

建议生成当前位于：

- `src/composables/use-analytics-derived.ts`
- `src/analytics/theme-documents.ts`

执行逻辑当前位于：

- `src/composables/use-analytics-interactions.ts`
- `src/analytics/orphan-theme-links.ts`

当前流程大致为：

1. 先识别主题文档。
2. 对每个孤立文档，计算它与主题文档之间的主题词命中次数。
3. 生成按 `matchCount` 排序的主题建议。
4. 用户点击建议后，向文档首段追加主题链接，或新增一段建议链接块。

### 1.2 当前方案的优点

- 速度快
- 规则简单
- 可解释性强
- 易于测试
- 与当前主题文档配置天然兼容

### 1.3 当前方案的明显局限

当前建议本质上仍是“基于词面匹配的主题建议”，因此会漏掉这些情况：

- 同义词、别名、术语变体不一致
- 标题和标签没有直接命中，但正文语义明显属于某主题
- 文档更适合补到某个核心文档，而不是主题文档
- 文档虽然当前孤立，但历史上与某些文档关系更强
- 某条补链虽然词面不强，但结构收益更高

因此，AI 的目标不应是替换现有规则，而应是在现有规则基础上补足语义召回和候选重排能力。

## 2. 目标场景

### 2.1 核心场景：孤立文档智能补链

用户在孤立文档详情中看到某篇文档后，希望系统回答：

- 这篇文档应该接回哪里
- 优先补到主题页，还是补到某个核心文档
- 为什么推荐这些目标
- 插入什么样的链接文案更自然
- 补完之后预计会带来什么变化

### 2.2 次级场景：弱连接文档补强

除了当前已经完全孤立的文档，还可以扩展到：

- 连接度很低的文档
- 当前沉没但历史有连接的文档
- 最近活跃但未进入主题网络的文档

这类文档虽然不一定在“孤立文档”列表中，但同样适合纳入 AI 补链建议体系。

### 2.3 未来扩展场景

同一套基础设施未来还可服务于：

- 出链不足核心文档的升级建议
- 缺主题页社区的建页建议
- 路径过长的捷径建议
- 分裂主题的互链建议

因此，本方案不应只做一个“单次 AI 调用”，而应设计成可复用的语义索引与候选重排系统。

## 3. 设计目标

### 3.1 目标

本方案应尽量同时满足以下目标：

1. 提升建议的全面性
2. 提升建议的准确性
3. 保持较低交互延迟
4. 复用现有分析结果
5. 保持可解释性
6. 保持渐进可落地

### 3.2 非目标

当前阶段不追求：

- 每次交互都让模型通读全部文档
- 用 AI 直接替代社区/桥接/传播节点分析
- 批量自动修改大量文档
- 让 AI 独立决定整个知识结构

## 4. 总体方案结论

推荐采用“三层架构”：

1. 结构分析层
2. 语义索引层
3. 在线推荐层

一句话概括：

`规则层与图分析层负责产生结构事实，语义索引层负责沉淀文档画像，在线推荐层负责召回候选并让 AI 做少量高价值重排。`

不建议采用以下两种极端方案：

- 每次都让 AI 通读全量文档后再给建议
- 只依赖一份摘要文本做全部判断

前者延迟高、成本高、稳定性差；后者信息损失大、容易误判。

## 5. 总体架构

### 5.1 架构分层

#### 第一层：结构分析层

直接复用当前已有结果：

- 主题文档
- 主题社区
- 核心文档
- 桥接节点
- 传播节点
- 孤立文档
- 沉没文档
- 路径与证据

这一层继续由现有规则与图分析实现产生，不交给 AI。

#### 第二层：语义索引层

为每篇文档增量生成“文档画像”，供推荐阶段快速使用。

建议画像字段包括：

- 摘要
- 关键词
- 主题标签候选
- 实体/术语
- 文档角色候选
- 向量 embedding
- 原文片段摘要索引
- 版本信息与缓存元数据

这一层的重点不是“让用户阅读”，而是“让系统高效召回和判断”。

#### 第三层：在线推荐层

在用户查看孤立文档或请求补链建议时：

1. 用结构结果和索引数据先召回候选
2. 控制候选数量
3. 仅对少量候选调用 AI 做重排与解释
4. 返回可直接绑定 UI 与动作的数据结构

### 5.2 推荐的数据流

#### 数据准备阶段

1. 采集当前分析样本中的文档原始信息。
2. 为新增或已变更文档生成语义画像。
3. 为画像写入私有索引库。
4. 为每个文档维护 embedding 与元数据。

#### 交互阶段

1. 用户打开孤立文档详情。
2. 系统从结构分析层拿到孤立文档列表。
3. 对当前孤立文档构造候选目标集合。
4. 对候选目标计算基础分。
5. 仅对前 N 个候选调用 AI 重排。
6. 返回推荐结果、解释理由、建议文案与预估收益。

#### 执行阶段

1. 用户确认推荐目标。
2. 系统执行插链。
3. 标记该文档的 AI 推荐缓存可失效。
4. 刷新当前分析结果。

## 6. 为什么不建议“每次先通读全部文档”

用户提出的方案是：

`让 AI 先通读一遍范围内的所有文档，生成摘要和关键词，后续分析仅根据摘要进行快速处理。`

这个思路有方向上的合理性，但不能直接照搬成运行时方案。

### 6.1 合理的部分

- 提前生成摘要、关键词、画像，确实能显著降低在线延迟
- 通过预处理沉淀语义信息，确实能让后续推荐更稳
- 这类派生数据很适合做增量缓存

### 6.2 不合理的部分

“后续分析仅根据摘要”并不适合本项目。

原因包括：

- 摘要是有损信息，容易丢掉关键术语
- 某些主题归属只在标题、路径或标签中明显
- 某些补链理由必须依赖局部原文片段
- 仅靠摘要很难判断链接应插成什么文案

因此，更合理的做法是：

`摘要只是画像的一部分，不是唯一依据。`

系统应综合使用：

- 摘要
- 关键词
- embedding
- 标签/标题/路径
- 当前结构分析结果
- 必要时的原文片段

## 7. 核心设计：文档画像

### 7.1 画像目标

文档画像用于支持：

- 候选召回
- 语义排序
- 推荐解释
- 文案生成
- 缓存复用

### 7.2 建议字段

建议为每篇文档维护如下索引记录：

```ts
interface DocumentSemanticProfile {
  documentId: string
  sourceUpdatedAt: string
  sourceHash: string
  profileVersion: number
  modelVersion: string

  title: string
  path: string
  hpath: string
  tags: string[]

  summaryShort: string
  summaryMedium: string
  keywords: string[]
  topicCandidates: string[]
  entities: string[]
  roleHints: string[]

  embedding: number[]

  evidenceSnippets: Array<{
    text: string
    score: number
  }>
}
```

### 7.3 字段说明

- `summaryShort`
  - 用于列表快速展示和初步判断
- `summaryMedium`
  - 用于更完整的语义判断
- `keywords`
  - 用于文本召回和解释
- `topicCandidates`
  - 用于辅助主题归属判断
- `entities`
  - 用于跨术语、别名和领域词汇匹配
- `roleHints`
  - 用于辅助判断它是主题页、案例页、方法页、草稿页等
- `embedding`
  - 用于语义相似度召回
- `evidenceSnippets`
  - 用于让模型在不读取全文的情况下看到关键片段

### 7.4 增量更新规则

建议按以下规则刷新画像：

- 文档 `updated` 变更
- 标题、路径、标签变更
- 模型版本变更
- `profileVersion` 升级
- 用户主动重建索引

如果上述条件都未变化，则直接复用缓存。

## 8. 核心设计：候选召回

### 8.1 总体策略

推荐结果的准确性，不能完全依赖 LLM。应先由系统召回较高质量的候选，再让 AI 排序。

建议采用“多路召回 + 合并去重”。

### 8.2 候选来源

建议至少包含以下召回通道：

#### 通道一：现有主题匹配规则

复用当前 `countThemeMatchesForDocument()` 结果，优点是：

- 快
- 稳
- 对主题文档场景有很强解释性

#### 通道二：embedding 相似召回

根据孤立文档与主题文档、核心文档、历史关联文档的向量相似度召回候选。

用于补足：

- 同义表达
- 术语不一致
- 标题命中弱但正文高度相关

#### 通道三：结构邻近召回

基于现有分析结果召回：

- 同社区 hub
- 传播节点
- 核心文档
- 历史连接文档
- 与其标签、路径或主题接近的活跃文档

#### 通道四：历史证据召回

如果该文档不是“从未有连接”，则优先纳入：

- 历史曾有连接的文档
- 最近一次连接过的主题文档
- 曾被同类文档共同引用的目标

### 8.3 候选过滤

在进入 AI 重排前，建议先过滤掉：

- 当前文档自身
- 已存在直接连接的目标
- 已被标记为不合适的人工否决目标
- 权重极低且语义相似度过低的目标

### 8.4 候选数量控制

建议控制为：

- 初步召回：30 到 80 个
- 规则打分后进入 AI：10 到 20 个
- 最终返回 UI：3 到 5 个

这是延迟、成本与效果之间较稳妥的平衡。

## 9. 核心设计：候选基础打分

在调用 AI 之前，建议系统先做一轮基础打分，用于：

- 压缩候选集合
- 提高 AI 输入质量
- 保持结果稳定

### 9.1 建议分数项

```ts
interface CandidateScore {
  documentId: string
  score: number
  themeMatchScore: number
  embeddingScore: number
  structuralScore: number
  historicalScore: number
  freshnessScore: number
  targetRoleScore: number
}
```

### 9.2 分数来源建议

- `themeMatchScore`
  - 现有关键词/主题名命中分
- `embeddingScore`
  - 语义相似度分
- `structuralScore`
  - 目标是否为主题文档、核心文档、hub、桥接相关节点
- `historicalScore`
  - 是否存在历史连接或相似连接模式
- `freshnessScore`
  - 目标近期是否仍活跃
- `targetRoleScore`
  - 目标是否更适合承担导航角色

### 9.3 目标类型优先级建议

建议优先把候选目标分为三类：

1. 主题文档
2. 核心/枢纽文档
3. 普通关联文档

默认情况下，若主题文档与普通文档得分接近，应优先推荐主题文档，因为它更符合当前产品的整理路径。

## 10. 核心设计：AI 重排与解释

### 10.1 AI 的职责

AI 在这套体系中的职责应限定为：

- 对高质量候选做语义判断
- 给出更符合人类整理习惯的排序
- 生成推荐理由
- 生成建议插入文案
- 粗略估计处理收益

AI 不应负责：

- 从全库无约束自由选目标
- 直接改写结构分析定义
- 不受约束地批量执行修改

### 10.2 AI 输入建议

每次请求建议只输入必要上下文，例如：

```ts
interface LinkSuggestionPromptInput {
  sourceDocument: {
    id: string
    title: string
    hpath: string
    tags: string[]
    summaryShort: string
    summaryMedium: string
    evidenceSnippets: string[]
  }
  graphContext: {
    isOrphan: boolean
    historicalReferenceCount: number
    communityHints: string[]
    selectedFilters: string[]
  }
  candidates: Array<{
    id: string
    title: string
    kind: 'theme-document' | 'core-document' | 'related-document'
    summaryShort: string
    keywords: string[]
    baseScores: Record<string, number>
    reasons: string[]
  }>
}
```

### 10.3 AI 输出建议

建议强制模型输出结构化结果：

```ts
interface LinkSuggestionResult {
  sourceDocumentId: string
  suggestions: Array<{
    targetDocumentId: string
    rank: number
    confidence: number
    recommendedAction: 'append-link' | 'insert-suggestion-paragraph'
    reason: string
    expectedBenefit: string
    draftText?: string
  }>
}
```

### 10.4 推荐的提示原则

提示词应明确要求：

- 优先选择更适合承担组织作用的目标文档
- 如果主题页明显合适，优先主题页
- 不要推荐与源文档语义过远的目标
- 解释必须结合给定候选和结构信号
- 不要发明不存在的文档

## 11. 核心设计：收益预估

为了提高用户接受度，建议为每条推荐给出简化版收益预估。

### 11.1 第一阶段建议的简化预估

不需要精确模拟，只需要给启发式结果：

- 补到主题文档后，预计移出孤立文档
- 目标主题社区规模预计增加
- 目标文档入链预计增加
- 当前主题网络覆盖更完整

### 11.2 后续可增强方向

后续可进一步增加：

- 路径长度变化预估
- 桥接风险下降预估
- 主题成熟度提升预估

## 12. 核心设计：建议执行

### 12.1 继续复用现有执行层

当前执行层已经具备：

- 插入建议链接
- 在首段追加链接
- 新建建议段落
- 同文档多主题建议合并
- 再次点击撤销

这些能力应尽量继续复用，而不是重写执行路径。

### 12.2 新增执行模式建议

现有执行层主要支持“直接插主题链接”。引入 AI 后，建议补两个执行模式：

1. `仅插链接`
2. `插入建议段落`

例如：

- 仅插链接：`((doc-theme-ai "主题-AI-索引"))`
- 建议段落：`可归入 AI 主题：((doc-theme-ai "主题-AI-索引"))`

第二种模式更适合 AI 生成自然文案。

### 12.3 人工确认策略

第一阶段建议全部采用：

- AI 给建议
- 用户点击确认
- 系统执行插入

不建议一开始就做自动批量补链。

## 13. 摘要与索引的存储设计

### 13.1 结论

默认不建议把摘要、关键词、embedding 等派生信息写回文档本身。

更合适的方式是：

- 存在插件私有索引库中

### 13.2 为什么不建议写回文档正文

原因包括：

- 污染用户原始笔记
- 影响同步与冲突管理
- 用户可能误改摘要或索引内容
- 这些信息本质上是可重建的派生数据
- 模型升级后需要批量重建，不适合和正文强绑定

### 13.3 为什么也不建议默认存成隐藏索引文档

隐藏索引文档比写回正文稍好，但仍有问题：

- 本质上还是进入了笔记库
- 可能被用户检索、误删、误同步
- 容易污染分析样本
- 大规模索引内容会让库中充满“不是笔记”的数据

### 13.4 推荐存储方案

推荐使用插件自己的私有存储，例如：

- 插件数据目录中的 `sqlite`
- 插件数据目录中的 `jsonl/json`

如果后续需要调试或导出，可额外提供：

- 重建索引按钮
- 导出调试视图
- 查看单篇画像的开发模式面板

### 13.5 什么时候允许落到文档本身

只有在以下场景中，才建议把 AI 结果写回文档：

- 用户主动确认“将该摘要保存为文档内容”
- 用户主动创建主题页草稿
- 用户主动插入建议段落

也就是“面向阅读与整理的产物”可以写回文档，“面向算法索引的派生数据”应留在私有索引库。

## 14. 数据存储结构建议

### 14.1 推荐表结构

如果使用 SQLite，可考虑以下表：

#### 表一：`document_semantic_profiles`

```sql
CREATE TABLE document_semantic_profiles (
  document_id TEXT PRIMARY KEY,
  source_updated_at TEXT NOT NULL,
  source_hash TEXT NOT NULL,
  profile_version INTEGER NOT NULL,
  model_version TEXT NOT NULL,
  title TEXT NOT NULL,
  path TEXT NOT NULL,
  hpath TEXT NOT NULL,
  tags_json TEXT NOT NULL,
  summary_short TEXT NOT NULL,
  summary_medium TEXT NOT NULL,
  keywords_json TEXT NOT NULL,
  topic_candidates_json TEXT NOT NULL,
  entities_json TEXT NOT NULL,
  role_hints_json TEXT NOT NULL,
  embedding_json TEXT NOT NULL,
  evidence_snippets_json TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

#### 表二：`document_link_suggestion_cache`

```sql
CREATE TABLE document_link_suggestion_cache (
  source_document_id TEXT NOT NULL,
  cache_key TEXT NOT NULL,
  model_version TEXT NOT NULL,
  suggestions_json TEXT NOT NULL,
  created_at TEXT NOT NULL,
  PRIMARY KEY (source_document_id, cache_key)
);
```

#### 表三：`document_index_jobs`

```sql
CREATE TABLE document_index_jobs (
  document_id TEXT PRIMARY KEY,
  status TEXT NOT NULL,
  retry_count INTEGER NOT NULL DEFAULT 0,
  last_error TEXT,
  scheduled_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

### 14.2 缓存键建议

建议缓存键至少包含：

- `documentId`
- `document.updated`
- 当前筛选上下文
- 当前时间窗口
- 当前主题文档版本
- 当前模型版本
- `profileVersion`

这样能避免旧结果污染新上下文。

## 15. 推荐流程设计

### 15.1 离线或增量索引流程

```text
文档变更检测
  -> 进入索引任务队列
  -> 提取原始字段
  -> 调用摘要/关键词/embedding 生成
  -> 写入私有索引库
  -> 标记画像可用
```

### 15.2 在线推荐流程

```text
用户打开孤立文档详情
  -> 读取当前结构分析结果
  -> 获取源文档画像
  -> 多路召回候选
  -> 基础打分与过滤
  -> 取前 10~20 个候选
  -> 调用 AI 重排与解释
  -> 返回前 3~5 个建议
  -> 用户确认后执行插链
  -> 刷新分析并失效相关缓存
```

## 16. UI 设计建议

### 16.1 在当前孤立文档详情中的最小改造

建议对每个孤立文档增加：

- `规则建议`
- `AI 建议`

其中：

- 规则建议立即可见
- AI 建议按需加载或懒加载

这样可以避免首次进入详情时全部等待 AI。

### 16.2 建议项展示字段

每条 AI 建议建议展示：

- 目标文档标题
- 目标类型
- 推荐理由
- 预估收益
- 置信度
- 一键插入
- 查看更多候选

### 16.3 失败与降级

如果 AI 不可用，界面应回退到：

- 当前规则建议
- 历史缓存建议

不要让补链入口整个消失。

## 17. 性能与延迟策略

### 17.1 延迟控制原则

延迟优化的关键不是“把模型做得更快”，而是：

- 减少在线读文规模
- 减少在线候选数量
- 减少不必要的重复调用
- 把能离线做的都提前做掉

### 17.2 具体策略

- 文档画像增量更新，不全量重算
- 先规则召回，再 AI 重排
- 仅对当前展开的孤立文档按需请求 AI
- 对推荐结果做短期缓存
- 对已确认插链的文档及时失效缓存

### 17.3 预取策略建议

可在以下时机做轻量预取：

- 孤立文档详情面板展开时
- 当前列表滚动接近可视区域时
- 用户停留某篇孤立文档超过一定时间时

## 18. 风险与应对

### 18.1 风险一：AI 推荐看起来合理，但实际不适合插链

应对：

- 强制 AI 只在候选列表内选择
- 提供理由和置信度
- 需要人工确认后执行

### 18.2 风险二：推荐变多，但解释不清

应对：

- 保留规则分数与结构信号
- 在 UI 上显示“为什么推荐”
- 支持查看原始候选来源

### 18.3 风险三：索引陈旧

应对：

- 使用 `sourceUpdatedAt + sourceHash`
- 变更后自动重建画像
- 提供“重建当前文档 AI 索引”入口

### 18.4 风险四：模型成本过高

应对：

- 摘要、embedding、推荐分成不同任务
- 优先缓存画像
- 在线阶段只做重排和文案生成

## 19. 分阶段实施建议

### 19.1 Phase 1：最小可用版本

目标：

- 在不改变现有主交互的前提下，补上 AI 重排能力

范围：

- 增加文档画像索引
- 增加 embedding 召回
- 在当前主题建议基础上增加 AI 重排
- 返回推荐理由与简单文案

不做：

- 大规模批量处理
- 自动创建复杂主题页
- 复杂收益模拟

### 19.2 Phase 2：增强版本

目标：

- 从“主题建议”升级为“补链建议”

范围：

- 候选目标不再只限主题文档
- 增加核心文档、历史目标、结构目标召回
- 增加收益预估
- 增加 AI/规则双视图对比

### 19.3 Phase 3：平台化版本

目标：

- 让同一索引能力服务更多 AI 整理场景

范围：

- 缺主题页社区自动建页
- 出链不足核心文档升级建议
- 路径过长优化建议
- 分裂主题互链建议

## 20. 建议的开发拆分

为了降低实施风险，建议拆成以下开发任务：

1. 定义私有索引存储结构
2. 实现文档画像生成任务
3. 实现 embedding 索引与相似召回
4. 实现候选合并、过滤与基础打分
5. 实现 AI 重排接口与结构化输出解析
6. 在孤立文档详情中接入 AI 建议视图
7. 复用现有执行层支持 AI 建议插入
8. 增加缓存、失效、降级与调试工具

## 21. 最终建议

本项目最合适的实现路径不是：

- 每次实时让 AI 通读全量文档
- 然后仅凭摘要做全部判断

而是：

`增量生成文档画像 + 利用现有结构分析召回候选 + 用 AI 对少量候选做语义重排与文案生成`

关于存储位置，建议明确区分两类数据：

- 可重建的派生索引数据：存插件私有索引库
- 面向用户阅读与整理的最终产物：在用户确认后写回文档

这条路线最符合当前项目的产品定位，也最有利于在准确率、延迟、成本和可解释性之间取得平衡。
