# LLM-Wiki 主题知识页生成方案

本文在 [文档索引与LLM-Wiki中间层重构方案](./文档索引与LLM-Wiki中间层重构方案.md) 的基础上，进一步明确：如何基于一组文档索引生成最终面向用户的 LLM-Wiki 页面。

核心前提已经确定：

- 文档索引是后台中间层，不是最终用户产品。
- LLM-Wiki 才是最终用户主要阅读的产物。
- Wiki 阶段始终只消费文档索引；索引缺失时，先生成索引，再生成 Wiki。
- Wiki 阶段不直接读取原文全文。

与前一版思路相比，本文的关键修正是：

- LLM-Wiki 不再默认采用单一“问题导向页面”模型。
- 页面本质上是“知识专题页”。
- 系统应根据主题类型和资料特征，自动选择合适模板，并按模块化章节生成页面。
- “问题导向”只是一种可选章节组织方式，而不是所有页面的统一母版。

## 1. 方案定位

LLM-Wiki 页面应定位为：

`面向学习、理解和持续扩展的知识专题页`

它不是：

- 结构分析报告页
- 引用网络诊断页
- 整理动作执行面板
- 单篇文档摘要页
- 统一按问答展开的页面模板

这意味着旧版模板中的以下章节不再作为 Wiki 固定主体结构：

- 结构观察
- 关系证据
- 整理动作

这些信息仍然有价值，但应退到后台，主要服务：

- 主题范围判定
- 来源文档排序
- 模板识别与章节启用
- 生成质量校验

换句话说，网络分析层继续参与“编排”和“约束”，但不再直接决定 Wiki 的可见页面骨架。

## 2. 已确认的关键决策

### 2.1 页面主类型

V1 的 LLM-Wiki 页面统一收敛为：

- 主题知识页

不单独生成“核心文档知识页”。如果用户从某篇核心文档触发 Wiki，也应先定位到所属主题，再刷新该主题的 Wiki 页面。

### 2.2 页面整体模型

页面采用：

- 中等共同底座
- 主模板自动选择
- 模块化章节启用

共同底座固定为：

- `主题简介`
- `关键信息 / 要点`
- `来源索引`

其余章节不固定死，而是由模板和资料特征共同决定。

### 2.3 模板选择方式

采用：

- 系统自动选择模板

不要求用户手动指定，也不在第一版让用户切换模板。

### 2.4 模板体系

V1 固定四类主模板：

- `技术专题`
- `软件产品使用方法`
- `社会话题`
- `书单 / 影单`

### 2.5 章节模块策略

页面不再默认携带“边界与分歧”“开放问题”“FAQ”等固定章节。

这些都属于：

- 条件触发模块

只有在当前资料确实支撑的情况下才启用。

### 2.6 来源标注粒度

采用：

- 混合级标注

具体规则：

- 默认按段落或知识点块集中标注来源
- 容易引起误解、信息密度高、具有争议性的句子，单独标来源

## 3. 目标与非目标

### 3.1 目标

- 基于多篇文档索引，生成面向用户的主题知识页。
- 让不同类型的主题使用不同模板，而不是统一硬套问答骨架。
- 让页面既可读、可学，又保留清晰来源锚点。
- 让生成结果更贴近原文命题，而不是概述性评价。
- 让页面可持续维护、可局部更新、可预览和可 diff。

### 3.2 非目标

- 当前阶段不做“原文全文直接进 Wiki prompt”的通读型流程。
- 当前阶段不做单文档知识页体系。
- 当前阶段不做完全自由拼装、无模板约束的页面生成。
- 当前阶段不做完全自动无人审阅写回。
- 当前阶段不把引用网络分析结果直接展示成 Wiki 的主体章节。

## 4. 页面模型

## 4.1 中等共同底座

所有模板共享的底座为：

1. `主题简介`
2. `关键信息 / 要点`
3. `来源索引`

这是所有页面共同的最小稳定结构。

### 4.1.1 主题简介

职责：

- 说明本页讨论的主题是什么
- 说明本页覆盖的范围
- 给出最直接的阅读入口

控制要求：

- 只写 1 段
- 不写空泛评价
- 不写“这是一篇介绍……”式套话

### 4.1.2 关键信息 / 要点

职责：

- 快速提取用户最需要先知道的核心信息
- 作为整页正文的高层摘要区

控制要求：

- 采用结构化要点，不写长篇散文
- 每条要点都应可追溯到来源
- 不允许写没有来源支撑的上位概括

### 4.1.3 来源索引

职责：

- 按文档分组列出本页主要来源
- 提供文档链接和必要的块链接
- 承担整页可追溯底座

来源索引不承担：

- 展示大量原文全文
- 替代正文表达

## 4.2 主模板

### 4.2.1 技术专题

适合：

- 概念
- 原理
- 架构
- 方法
- 协议
- 技术路线

高频模块：

- `核心原理`
- `实现或方法路径`
- `适用场景`
- `常见误解`

条件模块：

- `比较与选择`
- `开放问题`
- `争议点`

### 4.2.2 软件产品使用方法

适合：

- 软件功能
- 工作流
- 配置方法
- 上手教程
- 排障经验
- 常见问题

高频模块：

- `使用场景`
- `基本操作步骤`
- `进阶用法`

条件模块：

- `FAQ`
- `常见错误与排查`
- `比较与选择`
- `开放问题`

### 4.2.3 社会话题

适合：

- 公共议题
- 社会现象
- 政策讨论
- 价值争议
- 事件议题

高频模块：

- `主要观点或解释路径`
- `争议点`
- `典型案例或代表材料`

条件模块：

- `影响与后果`
- `开放问题`
- `FAQ`

### 4.2.4 书单 / 影单

适合：

- 作品清单
- 主题阅读 / 观看路径
- 作者或作品群介绍
- 选读 / 选看建议

高频模块：

- `作品地图`
- `代表作品`
- `观看 / 阅读顺序建议`
- `适合人群与使用场景`

条件模块：

- `比较与选择`
- `开放问题`
- `争议点`

## 4.3 跨模板通用模块

以下模块属于跨模板可复用模块：

- `使用场景`
- `核心原理`
- `实现或方法路径`
- `基本操作步骤`
- `进阶用法`
- `FAQ`
- `常见错误与排查`
- `主要观点或解释路径`
- `争议点`
- `开放问题`
- `典型案例或代表材料`
- `影响与后果`
- `作品地图`
- `代表作品`
- `观看 / 阅读顺序建议`
- `比较与选择`
- `常见误解`

其中：

- `FAQ`、`争议点`、`开放问题` 都不是保底章节
- 没有明确资料支撑的模块一律不生成
- 低置信度时优先回退到“共同底座 + 少量高确定模块”

## 5. 模板自动选择机制

## 5.1 判定流程

建议系统按三步完成模板选择：

1. `主题 bundle 诊断`
2. `主模板判定`
3. `模块启用判定`

模板选择和内容生成必须解耦，避免模型一边选结构一边写正文导致页面漂移。

## 5.2 判定信号

模板判定建议综合以下四类信号：

### 命题信号

看 `propositions` 主要在描述什么：

- “定义 / 原理 / 机制 / 架构 / 方法”偏 `技术专题`
- “操作 / 设置 / 步骤 / 报错 / 解决办法”偏 `软件产品使用方法`
- “观点 / 争议 / 背景 / 影响 / 立场”偏 `社会话题`
- “作品 / 作者 / 风格 / 推荐 / 顺序 / 适合谁”偏 `书单 / 影单`

### 证据块信号

看 `primarySourceBlocks` 与 `secondarySourceBlocks` 的文本形态：

- 教程型资料通常有明显步骤序列、菜单路径、配置项
- 社会议题更常出现案例、评价、冲突判断、群体差异
- 书单影单更常出现作品名单、作者名、题材、观看阅读建议

### 来源分布信号

看主题内文档是否高度同质：

- 大部分来源都在解释同一个工具怎么用，优先 `软件产品使用方法`
- 来源更像围绕同一议题的不同讨论材料，优先 `社会话题`

### 结构信号

看核心来源文档本身的结构倾向：

- 大量 FAQ、教程、配置记录，不要误判成技术专题
- 大量概念辨析、原理说明，不要误判成使用方法页

## 5.3 置信度与回退策略

模板选择建议分三档：

- `high`
- `medium`
- `low`

规则：

- 高置信度：主模板明确，允许启用较多模板专属模块
- 中置信度：主模板可判定，但模块要收紧
- 低置信度：仍自动选择最接近的主模板，但页面回退到“共同底座 + 少量高确定模块”

关键原则：

`低置信度时，宁可页面朴素，也不要为了像某种模板而硬补章节。`

## 5.4 关键边界规则

- 如果资料同时有“原理”和“步骤”，但“步骤 / 配置 / 避坑”占明显多数，优先 `软件产品使用方法`
- 如果资料围绕社会现象，主体是不同观点与解释路径，优先 `社会话题`
- 如果资料围绕推荐清单、进入路径和代表作品，优先 `书单 / 影单`
- `开放问题` 不参与主模板判定，它只是跨模板模块
- `争议点` 也不等于 `社会话题` 本身；技术专题同样可能有争议点，只是默认不开

## 6. 输入链路与数据打包

## 6.1 唯一语义输入源

Wiki 阶段只消费正文索引层数据：

- `positioning`
- `propositions`
- `primarySourceBlocks`
- `secondarySourceBlocks`
- 文档基础元信息

不消费：

- 补链语义档案
- 原文全文
- 向量嵌入

## 6.2 主题 bundle 构建

一页 Wiki 的直接输入不是“文档列表”，而是一个主题 bundle。

建议至少包含三层：

### 第一层：来源地图

按文档提供：

- 标题
- `positioning`
- 路径
- 标签
- 更新时间
- 文档在主题内的权重信息

### 第二层：命题层

按文档分组提供：

- `propositions`

### 第三层：证据层

按文档提供：

- `primarySourceBlocks`
- `secondarySourceBlocks`

## 6.3 分析层的后台作用

网络分析层和主题识别层仍然有价值，但其作用应退到后台：

- 决定本次主题范围
- 给来源文档排序
- 标记主题内核心来源与补充来源
- 为模板判定和模块启用提供辅助上下文

这些结构信号不直接变成页面章节，而是作为 prompt 辅助上下文。

## 7. 生成流程设计

不建议一次调用模型直接产出整页 markdown。推荐拆成 6 段：

1. 主题 bundle 构建
2. 模板诊断
3. 页面骨架规划
4. 逐章节生成
5. 开放问题与来源索引汇总
6. 本地渲染与引用绑定

## 7.1 主题 bundle 构建

由本地逻辑完成：

- 识别本次主题对应的来源文档
- 确保每篇来源文档都有可用索引
- 缺索引则先补索引
- 将索引整理成主题 bundle

## 7.2 模板诊断

由 LLM 执行，但只做分类与模块判定。

输出建议包含：

- `templateType`
- `confidence`
- `reason`
- `enabledModules`
- `suppressedModules`
- `evidenceSummary`

这一步不写正文。

## 7.3 页面骨架规划

模板选完后，不直接进入正文生成，而是先生成一个 `pagePlan`。

输出建议包含：

- `templateType`
- `confidence`
- `coreSections`
- `optionalSections`
- `sectionOrder`
- `sectionGoals`
- `sectionFormats`

这里决定这次页面到底生成哪些章节、顺序如何、每章的任务是什么。

## 7.4 逐章节生成

按章节类型分别生成内容，而不是整页一次输出。

建议支持 5 类章节生成格式：

- `overview`
- `structured`
- `qa`
- `debate`
- `catalog`

对应：

- `主题简介`、`关键信息 / 要点` 更偏 `overview` 或 `structured`
- `核心原理`、`适用场景`、`实现路径` 更偏 `structured`
- `FAQ` 使用 `qa`
- `争议点`、`开放问题` 更偏 `debate`
- `作品地图`、`代表作品` 更偏 `catalog`

这意味着“问题导向”不会消失，但它只适用于 `FAQ` 或少数适合问答展开的模块，而不再统治整页。

## 7.5 开放问题与来源索引汇总

在逐章节生成完成后，再统一处理跨章节结果：

- 收口当前资料中仍有争议或缺失的问题
- 整理整页来源索引
- 去除悬空引用和无支撑结论

开放问题的职责是：

- 显式列出当前资料尚无法稳定回答的问题
- 引导后续补资料和扩展主题知识库
- 防止页面伪装成“已经总结完毕”

它不是默认必开模块，只有在资料存在争议或缺口时才应出现。

## 7.6 本地渲染与引用绑定

由本地 renderer 完成：

- 将结构化结果转为稳定 markdown
- 生成最终章节顺序
- 插入来源上标
- 生成各章节下的来源注记
- 生成页尾来源索引

## 8. 数据结构建议

建议把生成结果统一成“页面计划 + 章节草稿”两层。

```ts
type WikiTemplateType =
  | 'tech_topic'
  | 'product_howto'
  | 'social_topic'
  | 'media_list'

type WikiSectionFormat =
  | 'overview'
  | 'structured'
  | 'qa'
  | 'debate'
  | 'catalog'

interface WikiTemplateDiagnosis {
  templateType: WikiTemplateType
  confidence: 'high' | 'medium' | 'low'
  reason: string
  enabledModules: string[]
  suppressedModules: string[]
  evidenceSummary: string
}

interface WikiPagePlan {
  templateType: WikiTemplateType
  confidence: 'high' | 'medium' | 'low'
  coreSections: string[]
  optionalSections: string[]
  sectionOrder: string[]
  sectionGoals: Record<string, string>
  sectionFormats: Record<string, WikiSectionFormat>
}

interface WikiSectionDraft {
  sectionType: string
  title: string
  format: WikiSectionFormat
  blocks: WikiContentBlock[]
  sourceRefs: string[]
}
```

## 9. 来源标注与链接策略

## 9.1 混合级引用

推荐规则：

- 默认按知识点块或段落块集中标来源
- 高风险句子、关键事实、容易失真的判断单独标来源

## 9.2 正文与链接分层

正文中只保留轻量编号，例如：

- `^1^`
- `^2^`

每个章节末尾放 `来源注记`，将编号映射为具体链接。

## 9.3 按章节类型选择文档链接与块链接

建议：

- `主题简介` 优先文档级来源
- `关键信息 / 要点` 适合按要点级引用
- `操作步骤`、`FAQ`、`常见错误与排查` 优先块级引用
- `争议点`、`开放问题` 必须保留更强的来源锚点，优先块级引用

## 9.4 不在 Wiki 正文中嵌入大段原文

虽然索引层中的“原文片段”会保留块级锚点，但 Wiki 正文不建议直接大量嵌入：

- `{{select * from blocks where id='...'}}`

嵌入块更适合开发期索引调试页，而不是最终对用户的 Wiki 正文。

## 10. 提示词设计要点

提示词设计不应再围绕“先规划问题，再逐题归纳”这一条单线展开，而应拆成三类 prompt。

## 10.1 模板诊断 prompt

目标：

- 判断最合适的主模板
- 决定应启用或抑制哪些模块

必须强调的约束：

- 不允许因为局部关键词就忽略整体主题
- 不允许为了完整而启用没有足够证据支撑的模块
- 若多个模板都像，只能选最贴近的一个，并通过收紧模块处理不确定性
- 最不能接受的失败不是“模板太保守”，而是“模板选错导致整页结构跑偏”

建议输出：

- `templateType`
- `confidence`
- `reason`
- `enabledModules`
- `suppressedModules`
- `evidenceSummary`

## 10.2 页面骨架规划 prompt

目标：

- 基于模板和模块，决定本次页面最终章节清单和顺序

必须强调的约束：

- 共同底座必须保留
- 非共同模块必须由当前资料支撑后才能进入页面计划
- 问答式章节不是默认主轴，只能在适合的模块中启用

建议输出：

- `sectionOrder`
- `sectionGoals`
- `sectionFormats`

## 10.3 逐章节归纳 prompt

目标：

- 按章节类型分别生成内容
- 保证内容贴近原文，不泛化、不拔高

必须强调的约束：

- 每个关键点必须能被一个或多个命题或证据块支撑
- 不允许写“这篇文章介绍了”“作者认为”“本文讨论了”这类摘要腔句式
- 不允许写评价性空话，例如“很系统”“很有启发”“值得借鉴”
- 不允许把边缘观点提升为主题共识
- 如果证据不足，不要补写，应直接收缩结论或抑制章节

建议提示词中的硬性术语：

- `忠于原文命题`
- `禁止评价式表述`
- `禁止无来源延展`
- `页面完整性低于内容可靠性`

## 10.4 开放问题汇总 prompt

如果将开放问题作为单独汇总步骤处理，提示词应强调：

- 只提取当前资料确实无法稳定回答的问题
- 优先选择对理解主题最关键的问题，而不是边缘细节
- 每个开放问题都要说明“为什么现在还不能下结论”
- 可以提示后续应补充的资料方向，但不能直接替用户回答
- 每个开放问题都必须绑定来源，证明它确实来自现有资料缺口或争议

## 10.5 通用系统约束要点

无论是模板诊断、骨架规划还是逐章节归纳，都建议反复强调以下系统级约束：

- 你的任务不是写宣传稿，而是整理一个可追溯的主题知识页。
- 你只能使用输入中给出的命题和证据块，不得调用外部常识补全主题。
- 如果输入无法支撑某个强结论，必须收缩结论，不得硬写完整答案。
- 输出必须是结构化 JSON；若无法满足字段要求，应返回失败原因，而不是输出半结构化 markdown。

## 11. 失败处理与降级策略

### 11.1 来源不足

若主题 bundle 中可用来源过少：

- 不生成正式 Wiki
- 只生成简版草稿或提示“当前主题资料不足”

### 11.2 模板低置信度

若模板判定置信度过低：

- 仍自动选择最接近的主模板
- 但页面回退到“共同底座 + 少量高确定模块”
- 不强行生成 `FAQ`、`争议点`、`开放问题` 等高假设章节

### 11.3 章节证据不足

若某个章节证据不足：

- 直接不生成该章节
- 不影响整页其他部分

### 11.4 引用绑定失败

若某条内容引用了不存在的来源 id：

- 该条内容不进入最终页面
- 或降级为文档级来源，而不是留下悬空编号

### 11.5 旧版索引

若发现输入索引仍是旧版结构：

- 直接删除旧索引并重建
- 不做迁移兼容

## 12. 测试重点

建议重点覆盖：

- 模板是否正确识别
- 低置信度时是否正确回退到保守骨架
- 索引缺失时是否先补索引
- 页面骨架规划是否符合模板与模块启用规则
- `FAQ`、`争议点`、`开放问题` 是否只在有足够证据时启用
- 逐章节内容是否由命题和证据块支撑
- 问答式结构是否只在适合的模块出现，而不是污染整页
- 混合级引用渲染是否正确
- 文档链接与块链接的选择是否符合规则
- 旧版索引是否被直接删除并重建

## 13. 后续扩展主题知识库的建议

主题知识库不应只靠“生成更多 Wiki 页面”扩展，而应围绕资料质量、主题边界和页面拆分逐步做厚。建议优先从以下几个方向推进：

### 13.1 围绕开放问题反向补资料

最值得补的资料，不是随机新增同主题文章，而是直接围绕 Wiki 中沉淀下来的开放问题去找。

### 13.2 补“同主题不同层次”的资料

扩展目标不是简单增加数量，而是补齐层次结构，例如：

- 方法拆解
- 条件与前提
- 反例与失败案例
- 具体案例
- 更高层抽象总结

### 13.3 形成“主题主页 + 子主题页”结构

当某个主题的关键内容已经明显膨胀，单页难以承载时，应自然拆出子主题，而不是继续压回一页。

### 13.4 为主题页维护“长期未解问题”

开放问题不只是本次生成的尾部章节，也可以作为长期知识治理入口。

### 13.5 后续增加“比较型主题”

当主题知识库积累到一定规模后，后续不应只生成“单主题总结页”，还可以增加比较型页面，例如：

- 两种方法的差异
- 两类概念的边界
- 不同来源对同一问题的解释路径比较

## 14. 与现有方案的关系

本方案与 [文档索引与LLM-Wiki中间层重构方案](./文档索引与LLM-Wiki中间层重构方案.md) 的关系是：

- 前者回答“单文档索引应该长什么样”
- 本文回答“主题 Wiki 应该怎样消费这些索引并生成多模板知识页面”

两者共同组成新的完整链路：

`证据块 -> 原文命题 -> 单文档底稿 -> 主题 bundle -> 模板诊断 -> 页面骨架规划 -> 逐章节生成 -> LLM-Wiki 页面`

## 15. 最终建议

推荐将 LLM-Wiki 明确定位为：

- 主题级知识编译页
- 以中等共同底座保证稳定性
- 以自动选择主模板保证贴题性
- 以模块化章节保证灵活性
- 以混合级来源标注保证可追溯
- 以文档索引为唯一语义输入层

这意味着项目后续不应再把 Wiki 做成：

- 结构分析面板的页面化复刻
- 摘要的再摘要
- 所有主题统一按问答方式展开的页面
- 原文片段的大量拼贴

而应把它做成真正可读、可学、可扩展、可追溯的主题知识页。

这与“脉络镜”当前方向是一致的：

`不是直接把分析结果写成报告，而是把索引编译成用户真正会长期阅读和复用的知识专题页。`
