# LLM Wiki Chat Mention Popup Design

## 背景

LLM Wiki 对话窗口当前支持在输入框中输入 `@` 以切换来源 wiki 页面，但现有交互存在两个问题：

1. 提及候选列表以普通块级元素插入在消息区与输入区之间，窗口高度不足时会被挤压和遮挡。
2. 候选列表没有随着 `@` 后输入的关键词实时筛选，用户无法通过继续输入快速缩小范围。

本次设计只处理 LLM Wiki 对话中的 `@` 提及浮层，不改 AI 路由、消息保存格式、Markdown 渲染和来源切换文案。

## 目标

1. 当用户在聊天输入框中输入 `@` 时，显示可用 wiki 页候选。
2. 当用户在 `@` 后继续输入关键词时，只按 wiki 页面标题做不区分大小写的包含匹配。
3. 候选浮层在窗口高度不足时仍可见，不再被输入区或底部区域挤压遮挡。
4. 候选浮层最多显示 3 条可见结果；若匹配数超过 3 条，底部额外显示 `...` 提示还有更多结果，并允许列表区域滚动。
5. 保持现有来源切换语义：用户可通过键盘、鼠标或直接发送带 `@` 的消息完成来源切换。

## 非目标

1. 不根据 wiki 页面摘要、正文或其他字段做提及匹配。
2. 不实现 textarea 光标级精确定位浮层。
3. 不改动多轮聊天的路由策略、消息结构或 AI 请求流程。
4. 不新增“无匹配结果”占位提示。

## 当前实现概览

### 相关文件

- `src/components/WikiChatDialog.vue`
  - 负责输入框 `@` 检测、候选弹窗渲染、键盘选择与样式。
- `src/composables/use-wiki-chat-session.ts`
  - 已维护 `mentionFilter`、`mentionPopupVisible`、`filteredPages`、`switchSource()` 等状态与行为。
- `src/components/WikiChatDialog.test.ts`
  - 当前仅覆盖 Markdown 渲染接入与部分样式断言。
- `src/composables/use-wiki-chat-session.test.ts`
  - 当前未覆盖提及过滤与候选显示行为。

### 已知根因

1. `use-wiki-chat-session.ts` 已有 `mentionFilter` 与基于标题包含匹配的 `filteredPages`，但 `WikiChatDialog.vue` 的 `handleInput()` 只控制 `mentionPopupVisible`，没有把 `@` 后文本同步到 `mentionFilter`，导致候选列表不会随输入变化。
2. `WikiChatDialog.vue` 中的提及候选列表位于输入区之前，参与正常文档流布局，所以当窗口高度不足时，它会被有限的剩余空间压缩甚至遮挡。

## 设计方案

### 方案选择

本次采用“输入区上方绝对定位浮层 + 最多可见 3 条 + 超出时显示 `...` + 内部滚动”的方案。

不采用光标级精确定位方案，原因是需要额外维护 textarea 光标像素坐标，复杂度明显高于实际收益。

### 交互设计

1. 输入框中仅识别“光标前最近一次提及片段”，匹配规则保持为光标前文本末尾的 `@关键词`。
2. 当检测到 `@` 或 `@关键词` 时：
   - 把关键词实时写入 `mentionFilter`。
   - 若存在匹配结果，则显示提及浮层。
   - 若没有匹配结果，则隐藏提及浮层。
3. 当用户只输入 `@` 而没有关键词时，显示默认候选列表。
4. 当匹配结果数量大于 3 时：
   - 浮层列表区域最多展示 3 条候选项高度。
   - 允许列表区域内部滚动查看更多项。
   - 浮层底部额外显示一条不可选的 `...` 提示还有更多结果。
5. 键盘交互：
   - `ArrowDown` / `ArrowUp` 在实际候选项之间移动高亮。
   - `Enter` 选中当前高亮候选项。
   - 若没有候选项，则保留原有发送逻辑。
6. 鼠标交互：点击候选项直接切换来源页。
7. 选中候选项后：
   - 移除输入框中的 `@关键词`。
   - 关闭提及浮层。
   - 写入现有“来源已切换”的 system message。
   - 保持输入框焦点。
8. 若用户不从浮层显式选择，而是直接发送包含 `@关键词` 的消息，发送前仍按现有逻辑尝试用 `@关键词` 匹配标题并切换来源页。

## 视觉与布局设计

1. 提及浮层改为锚定在输入区容器内的绝对定位元素，显示在输入区上方。
2. 浮层不再占用消息区与输入区之间的正常布局高度，而是覆盖在消息区下沿之上。
3. 浮层需要具备以下视觉属性：
   - 明确的边框与背景，保持与现有聊天面板风格一致。
   - 独立阴影，确保在消息区上方有清晰层次。
   - 限定最大高度为 3 条候选项加 `...` 提示行的高度。
4. 候选项标题过长时保持单行省略，避免浮层高度被单条内容撑开。
5. `...` 提示为静态行，不参与 hover、高亮或回车选择。

## 数据与状态设计

### 会话层

`src/composables/use-wiki-chat-session.ts` 继续作为提及数据的单一来源：

- `mentionFilter`
  - 保存当前 `@` 后的关键词。
- `filteredPages`
  - 保持按 `wikiPages[].title` 做不区分大小写的包含匹配。
  - 不再在这里截断到 8 条，避免组件层无法知道是否还有更多结果。
- `mentionPopupVisible`
  - 保持由输入行为驱动，但只有在存在匹配结果时才展示。

为避免组件直接改写多个 `ref` 造成重复逻辑，会话层需要暴露一个明确的输入同步入口，例如“根据光标前文本更新 mention 状态”的方法，供组件在 `input` 事件中调用。

### 组件层

`src/components/WikiChatDialog.vue` 基于会话层状态派生仅用于渲染的视图状态：

- `visibleMentionPages`
  - `filteredPages` 的前 3 项。
- `hasMoreMentions`
  - `filteredPages.length > 3`。

键盘高亮索引仅针对完整 `filteredPages` 维护，但渲染时默认展示前 3 条，并通过浮层内部滚动让更多结果可见。组件需要在高亮项超出当前滚动区域时自动滚动到可见区域内。

## 错误与边界处理

1. 如果 `wikiPages` 为空，则不显示提及浮层。
2. 如果 `@` 出现在光标后方但不在光标前最近片段中，不触发提及浮层。
3. 如果输入中包含多个 `@`，只处理光标前最近的那个片段。
4. 如果用户删除了 `@` 或在 `@` 后输入空白，立即关闭提及浮层并清空 `mentionFilter`。
5. 如果匹配结果为空，隐藏浮层，不显示空态提示。
6. 如果通过键盘选择时高亮索引越界，应在筛选结果变化后自动归位到首项。

## 测试设计

### `src/composables/use-wiki-chat-session.test.ts`

新增或调整以下测试：

1. `mentionFilter` 为空时，`filteredPages` 返回全部 wiki 页面，不做标题外字段匹配。
2. 设置 `mentionFilter = '机器'` 时，只返回标题包含“机器”的页面。
3. 匹配忽略大小写。
4. 没有标题匹配时，`filteredPages` 返回空数组。
5. 通过会话层新增的 mention 同步入口更新输入时，能正确同步 `mentionFilter` 与 `mentionPopupVisible`。

### `src/components/WikiChatDialog.test.ts`

新增或调整以下测试：

1. 输入 `@` 时渲染提及浮层。
2. 输入 `@机器` 时，仅展示标题包含“机器”的 wiki 页面。
3. 匹配结果超过 3 条时：
   - 浮层只在初始可见区域展示 3 条高度。
   - 底部显示 `...` 提示。
4. 浮层样式使用绝对定位并挂在输入区容器内，而不是普通文档流块级布局。
5. 键盘上下移动与回车选择行为仍然可用。

## 验收标准

1. 在较低聊天窗口高度下，输入 `@` 后的提及浮层仍完整可见，不被输入区或底部区域遮挡。
2. 在输入 `@关键词` 时，候选结果随输入即时变化，并且仅依据 wiki 页面标题做包含匹配。
3. 匹配结果超过 3 条时，浮层出现内部滚动能力，并显示 `...` 提示还有更多结果。
4. 通过键盘或鼠标选择候选项后，来源页切换行为与当前系统消息语义保持一致。
5. 现有聊天发送、来源切换和消息保存能力不回归。
