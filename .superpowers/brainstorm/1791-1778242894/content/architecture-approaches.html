架构方案对比

三种实现路径，各有侧重

A

Composable + 重构 Dialog（推荐）

新建 use-wiki-chat-session.ts 管理消息历史、来源追踪、路由状态、上下文构建。WikiChatDialog.vue 作为 UI 壳层负责渲染和交互。

✔ 符合项目现有模式（composable 管逻辑，组件管 UI）
✔ 逻辑可独立测试
✔ 组件职责清晰，不会膨胀
⚠ 新增 1 个 composable 文件
B

全部内联到 Dialog

在 WikiChatDialog.vue 内部完成所有改造：消息数组、路由逻辑、@ 检测、上下文构建。不新建文件。

✔ 文件数量最少
✔ 改动集中，容易追踪
⚠ 组件将膨胀到 400+ 行，混合 UI 和业务逻辑
⚠ 难以单独测试路由和上下文逻辑
C

通用 ChatWindow 组件 + Wiki 适配层

新建通用 ChatWindow.vue（气泡、输入、@提及、头像），WikiChatDialog 通过 props/events 适配 Wiki 特有逻辑（路由、来源管理）。

✔ UI 组件可复用
✔ 职责分离最彻底
⚠ 当前项目无其他聊天场景，可能过度抽象
⚠ 抽象层较多，调试链路变长
推荐方案 A

项目已有的 use-analytics-llm-wiki-chat.ts 正是 composable 管理状态的模式。 方案 A 自然延续这一风格：use-wiki-chat-session.ts 管理消息历史和路由， WikiChatDialog.vue 只负责 UI 渲染和用户交互。 逻辑可测试、组件不膨胀、改动范围可控。
